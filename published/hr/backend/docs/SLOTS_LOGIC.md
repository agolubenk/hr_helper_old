# Логика определения слотов (календарь → доступные слоты)

Документ описывает, как из событий календаря получаются «доступные слоты» для скринингов и интервью. Используется для выравнивания кода и исправления ошибок.

---

## 1. Назначение

- **Слот** — непрерывный свободный интервал времени, в который можно поставить встречу заданной длительности (скрининг или интервью).
- Слоты считаются по календарям: пользователя (рекрутера), компании, при интервью — ещё по календарям выбранных интервьюеров.
- Событие календаря = «занято»: в этот интервал слот недоступен (с учётом настройки «игнорировать не принятые встречи» и интервала между встречами).

---

## 2. Источники событий

| Контекст | Календари | Примечание |
|----------|-----------|------------|
| Скрининги (chat_workflow) | Пользователь + компания | Один список событий для расчёта |
| Интервью, загрузка страницы (chat_workflow) | Пользователь + компания | Базовые слоты; слоты с учётом интервьюеров — через API (в зависимости от выбранного варианта в настройках компании учитываются слоты всех встреч (учитывая не принятые) или только принятых встреч)|
| Интервью, API `/api/interview-slots/` | Пользователь (если include_user) + компания + выбранные интервьюеры (в зависимости от выбранного варианта в настройках компании учитываются слоты всех встреч (учитывая не принятые) или только принятых встреч)| По каждому участнику — свой календарь |
| Третья неделя (API) | Аналогично, days_ahead=21 | Отдельный запрос по типу встречи и настройкам компании |

Формат одного события (после приведения к виду для калькулятора):

- `start` — начало (строка ISO или dict с `dateTime`/`date`)
- `end` — конец (аналогично)
- `is_all_day` / `isallday` — событие на весь день (такие не блокируют почасовые слоты в текущей логике)
- Для фильтрации по ответу участника нужны `attendees`: список с `email` и `responseStatus` (или `response_status`).

---

## 3. Настройка компании «Игнорировать не принятые встречи»

**Где хранится:** `CompanySettings.ignore_non_accepted_meetings` (bool).

- **True (галочка «Игнорировать не принятые встречи» = ДА)**  
  Событие считается **занятым**, если пользователь (или интервьюер) в участниках и его ответ **не** `declined`.  
  То есть: `accepted`, `needsAction`, `tentative` — всё блокирует слот.

- **False (галочка = НЕТ)**  
  Событие считается **занятым** только если ответ участника **`accepted`**.  
  То есть: `needsAction`, `tentative` — слот **свободен**, пока не принято.

Правило по статусам:

- `declined` — всегда не блокирует слот (участник не участвует).
- `cancelled` / `declined` у самого события (status) — событие целиком пропускаем.

---

## 4. Фильтрация событий перед расчётом слотов

Перед передачей списка событий в калькулятор слотов применяется фильтр по участнику (рекрутер или интервьюер).

Для **каждого** события:

1. Если `status` события `cancelled` или `declined` — **исключить** из списка.
2. Найти участника по `email` (текущий пользователь или интервьюер).
3. Взять его `responseStatus` (или `response_status`), по умолчанию `needsAction`.
4. Если статус `declined` — **исключить** событие.
5. Если настройка **«не игнорировать»** (`ignore_non_accepted_meetings == False`) и статус **не** `accepted` — **исключить** событие (слот считаем свободным).
6. Иначе событие **включаем** в список занятости (блокирует слот).
7. Важно получать также не принятые, но потенциальнаые события/встречи, которые используются в анализе слотов. Они могут быть от других людей и быть в статусе не принятых.

Если у события нет участника с данным email (например, событие календаря компании без пользователя в attendees):

- При **«игнорировать» = ДА** — включаем (считаем занятым).
- При **«не игнорировать» = НЕТ** — в текущей реализации такие события попадают в список «consider» только если явно не найдены в attendees; лучше явно зафиксировать: события без текущего участника в attendees считать занятыми (календарь компании/ресурс).

---

## 5. Два варианта списков (для кэша и выбора по настройке)

Чтобы не пересчитывать при смене настройки, формируются **два** списка событий из одного набора `calendar_events_data`:

- **ignore_non_accepted** — вариант «игнорировать = ДА»: в список входят все события, кроме отменённых и тех, где участник `declined`.
- **consider_non_accepted** — вариант «не игнорировать = НЕТ»: в список входят только события, где участник `accepted` (или участника нет в событии — см. п. 4).

Один проход по событиям, один раз строится пара списков. По текущей настройке компании выбирается один из них и передаётся в калькулятор.

---

## 6. Кэш

- **Что кэшируется:** объект `{ 'ignore_non_accepted': [...], 'consider_non_accepted': [...] }` — оба варианта списков событий в формате для калькулятора (`start`, `end`, `is_all_day`). Каждый список — события, которые считаются «занятыми» при соответствующей настройке компании.
- **Ключ:** `slot_events_both_variants`, user_id, scope=`chat_workflow_14`, days_ahead=14. TTL задаётся в `settings.API_CACHE_TIMEOUT['slot_events_both_variants']` (180 сек).
- **Когда читаем кэш:** при формировании слотов на странице chat_workflow, если **не** передан параметр принудительного обновления (`refresh_slots` / `force_refresh`).
- **Когда пишем кэш:** только при пересчёте обоих вариантов в chat_workflow — при промахе кэша или при принудительном обновлении (после пересчёта оба списка сохраняются).
- **Принудительное обновление:** при наличии в GET-запросе `refresh_slots` или `force_refresh` кэш **не читаем**, всегда пересчитываем оба варианта из актуальных `calendar_events_data`, записываем в кэш, отдаём страницу; затем редирект на тот же путь без параметров `refresh_slots` и `force_refresh`, чтобы в URL не оставался флаг. Кнопка «Обновить слоты» на фронте добавляет `refresh_slots=1` к URL и перезагружает страницу.
- **Инвалидация:** при вызове `GoogleAPICache.invalidate_user_cache(user_id)` сбрасывается и кэш слотов для этого пользователя.

## 7. Калькулятор слотов (SlotsCalculator)

**Файл:** `logic/slots_calculator.py`, класс `SlotsCalculator`.

### Входные данные

- **Список событий** — каждый элемент с полями:
  - `start` — начало: ISO-строка (с `T` для даты-времени), либо dict с ключом `dateTime` или `date`, либо объект `datetime`.
  - `end` — окончание (аналогично).
  - `is_all_day` или `isallday` — событие на весь день (такие не создают почасовых занятых интервалов в текущей реализации).
  - Опционально `title` / `summary` — для исключения обедов и отпусков (см. ниже).
- **Параметры конструктора:** `work_start_hour` (по умолчанию 10), `work_end_hour` (18), `meeting_interval_minutes` (15). Часовой пояс внутри калькулятора — `Europe/Minsk`. Откуда берутся значения — см. п. 7.1.

### Выход

- **Методы:** `calculate_slots_for_week(events_data, required_duration_minutes, week_offset=0)` и `calculate_slots_for_two_weeks(events_data, required_duration_minutes)`. Два недели — возвращается **до 10** дней (оставшиеся дни текущей недели + следующая неделя). В субботу и воскресенье «текущей» считается следующая неделя, затем неделя через одну, чтобы всегда были слоты для выбора.
- **Формат дня:** `{ 'date': ISO-дата, 'dateStr': 'ДД.ММ.ГГГГ', 'weekday': 'Пн'|'Вт'|…|'Пт', 'meetingsCount': int, 'availableSlots': str }`.
- **availableSlots** — строка вида `"11.00-12.00, 14.00-15.30"` или `"Нет свободных слотов"`. Время в формате `H.MM`.

### Внутренняя логика

1. **Рабочие дни:** только пн–пт. Начало недели — понедельник. **Прошедшие** дни пропускаются; **текущий день** показывается, слоты для него — от текущего момента. В субботу и воскресенье «текущей» считается следующая неделя (см. два недели).
2. **Фильтр по дате:** `_filter_events_for_date` оставляет события, у которых `_parse_event_start` попадает в заданный день (date_start..date_end). Время парсится в ISO; при передаче datetime приводится к Europe/Minsk.
3. **Занятые интервалы:** для каждого события дня (не на весь день, с валидными start/end) интервал расширяется на `meeting_interval_minutes` минут до и после; пересечение с рабочими часами добавляется в `occupied_intervals`. События на весь день не создают занятых интервалов. Интервалы сортируются и объединяются при пересечении.
4. **Свободные интервалы:** между концом рабочего дня «вчера» и первым занятым, между занятыми и после последнего занятого до конца рабочего дня. Для **сегодняшнего дня** старт поиска свободных — текущее время (без секунд), для будущих дней — начало рабочего дня.
5. **Минимальная длина слота:** свободный отрезок попадает в вывод только если его длина ≥ max(15, required_duration_minutes) минут.
6. **Исключения из «встреч»** (для подсчёта `meetingsCount`; занятость дня считается по всем событиям, включая на весь день): события на весь день не считаются «встречей» в счётчике; по `title`/`summary` исключаются события, содержащие «обед», «lunch», «отпуск», «vacation», «выходной», «day off».

### 7.1. Источники рабочих часов (work_start_hour, work_end_hour)

Единая логика вынесена в сервис **`logic/working_hours_service.py`**, функция `get_effective_working_hours(user, company_settings=None, user_is_participant=True)`. Возвращает `dict` с ключами `work_start_hour`, `work_end_hour`, `meeting_interval_minutes`.

**Приоритет (слоты пользователя > слоты компании > дефолт):**

1. **По умолчанию:** `work_start_hour=10`, `work_end_hour=18`.
2. **Слоты компании:** если в `CompanySettings` заданы `work_start_time` и `work_end_time` — переопределяют дефолт.
3. **Слоты пользователя:** если `user_is_participant=True` и у пользователя заданы `interview_start_time` / `interview_end_time` — переопределяют слоты компании (наивысший приоритет). Если пользователь не участник встречи (`user_is_participant=False`) — слоты пользователя не учитываются, используются только слоты компании или дефолт.

`meeting_interval_minutes`: при `user_is_participant=True` берётся из пользователя (`user.meeting_interval_minutes`), иначе 15.

В **API слотов интервью** (`api_interview_slots`) передаётся `user_is_participant=include_user`: когда рекрутер не выбран участником встречи (`include_user=0`), для расчёта слотов используются только часы компании или значения по умолчанию.

Использование сервиса: `api_interview_slots` (с флагом `user_is_participant=include_user`), third week API, `chat_workflow` (слоты при загрузке), `google_oauth.models` (расчёт слотов по дням), `enhanced_datetime_parser._get_user_business_hours`.

### Парсинг времени

- `_parse_event_start` / `_parse_event_end`: поддерживаются dict с `dateTime` или `date`, строка (ожидается наличие `'T'` для даты-времени), объект `datetime`. Строка парсится через `datetime.fromisoformat`, `Z` заменяется на `+00:00`; при отсутствии таймзоны подставляется UTC; результат переводится в `Europe/Minsk`. При ошибке парсинга возвращается `None`, такое событие не создаёт занятый интервал.

## 9. Важные краевые случаи

- **Событие без attendees или без текущего пользователя в attendees:** в `_build_slot_events_both_variants` такое событие попадает в оба списка (ignore и consider), т.е. считается занятым. Это подходит для событий календаря компании/ресурса, где пользователь не указан как участник.
- **Формат `start`/`end`:** калькулятор принимает ISO-строку или dict с `dateTime`/`date`. В chat_workflow в `calendar_events_data` после подготовки для шаблона поля уже в формате, пригодном для калькулятора (ISO-строки). В кэше хранятся те же структуры (`start`, `end`, `is_all_day`).
- **Часовой пояс:** все времена в калькуляторе приводятся к Europe/Minsk. События из API при подготовке в chat_workflow конвертируются в Minsk; при передаче в API interview-slots парсинг и конвертация выполняются в view. Необходимо единообразие: либо все входные события в UTC/ISO с таймзоной, либо явная интерпретация.
- **Пустой список событий:** калькулятор возвращает слоты по рабочим часам (весь день свободен в пределах рабочих часов и требуемой длительности).
- **События на весь день:** не участвуют в расчёте занятых интервалов (не создают почасовых блоков); в `meetingsCount` не учитываются (через `_is_meeting`).
- **Ошибка парсинга даты события:** `_parse_event_start`/`_parse_event_end` возвращают `None`, событие не создаёт занятый интервал и не попадает в фильтр по дате при отсутствии даты.

## 10. Чек-лист при исправлениях

- [ ] Настройка `ignore_non_accepted_meetings` везде читается одинаково: True = «игнорировать не принятые» (слот занят при accepted/needsAction/tentative), False = только accepted блокирует слот.
- [ ] События со статусом `cancelled`/`declined` и события, где участник в статусе `declined`, везде исключаются из списка «занятости».
- [ ] При «не игнорировать» из списка занятости исключаются события, где участник не `accepted` (needsAction, tentative).
- [ ] Два варианта списков (ignore_non_accepted / consider_non_accepted) строятся один раз в `_build_slot_events_both_variants` и согласованы с п. 3–5 документа.
- [ ] Кэш хранит оба варианта; при `refresh_slots` или `force_refresh` в GET кэш не читается, данные пересчитываются и записываются в кэш; после отдачи страницы выполняется редирект без этих параметров.
- [ ] Формат событий, передаваемых в `SlotsCalculator`, совпадает с поддерживаемым парсерами: `start`/`end` — ISO-строка с `T` или dict с `dateTime`/`date`; при необходимости конвертация в Europe/Minsk выполняется до передачи или внутри калькулятора.
- [ ] В api_interview_slots и third week при недоступности CompanySettings используется безопасное значение (например getattr(..., True)), чтобы не падать и трактовать как «игнорировать не принятые».
- [ ] Рабочие часы и интервал для калькулятора везде получаются через `logic.working_hours_service.get_effective_working_hours`: приоритет — пользователь (interview_start_time, interview_end_time) > компания (work_start_time, work_end_time) > дефолт (10, 18, 15). При `user_is_participant=False` — только компания или дефолт.
